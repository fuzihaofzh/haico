# Documentation System

HAICO 文档体系的元文档。定义文档分层模型、更新规则、维护机制。
**每次更新任何文档前必须读本文件。**

---

## 三层模型

文档按抽象程度分三层，外加一层需求规格：

```
L-1  prd/                 需求规格 — 功能做什么（与代码生命周期解耦）
L0   AGENTS.md            理念与流程引导 — 为什么这样设计 + 怎么用文档
L1   architecture 契约段   概念与约束 — 概念表 + 规则/契约/依赖方向
L2   architecture 实现段   具象实现 — 模式/步骤/目录/调试
```

阅读链条：**理念（方向）→ 概念（是什么）→ 约束（不能怎么做）→ 实现（怎么做）**。
每一层是上一层的具象化，不跳跃。

### 各层定位

| 层 | 内容性质 | AI 何时读 | 更新触发 | 准入判据 |
|---|---|---|---|---|
| L-1 | 需求规格 | 理解功能为什么存在时 | 需求变更 | 有独立产品价值 |
| L0 | 设计理念 + 文档使用流程 | 每次开始工作前（常驻） | 理念诞生/消亡 | 极少，几条 |
| L1 | 概念定义 + 约束规则 | 改动涉及某概念/约束时 | 契约被代码推翻/扩展 | 跨子系统影响 |
| L2 | 实现模式与步骤 | 动手写具体代码时 | AI 照文档做跑不通时 | 单子系统内 |

### L0 理念的准入判据

L0 只承载方向性陈述，不承载可操作的具体内容。判据：
- 该理念指导**全局**行为，不限于单个子系统
- 不可操作——读完不知道具体怎么做，只知道方向
- 几乎不变——理念比规则稳定

**不进 L0 的内容**：概念定义、约束规则、接口表、目录结构、实现模式。
这些全下沉到 L1/L2。

### L1 概念的准入判据

L1 概念表中的每个概念必须满足：
- 在 **≥2 个子系统**中被使用或引用
- 只在单子系统内使用的概念，在 L1/L2 首次出现时定义，不进全局概念表

术语统一规则：L1/L2/代码中引用 L0 概念时，用概念表中的名称，不重新定义、不用别名。
需要展开时写"Adapter（见 AGENTS.md §Design Philosophy）"。

### L1 与 L2 在同一文件内的分界

同一篇 `architecture/*.md` 文件内，L1（契约段）和 L2（实现段）共存但用 heading 显式分界。
分界判据：

- **L1 契约段**：描述跨子系统的规则、契约、依赖方向、接口签名、概念表。
  读者是"改某子系统但需要懂相邻子系统契约的人"。
- **L2 实现段**：描述单子系统的实现模式、目录结构、迁移清单、调试技巧、代码示例。
  读者是"动该子系统具体文件的人"。

AGENTS.md 的 Doc Index 表标注每篇文档的哪些 section 属于 L1、哪些属于 L2。

## 更新红线

更新文档时不可违反的硬约束：

### 红线 1：禁止跨层复制规则内容

同一条规则只能在一个层有**完整内容**。L0 可有理念陈述（一句话），
L1 有约束详情（背景+例外+检测），不能两边都写完整论证。

- 违反后果：改一处忘另一处 → 漂移 → AI 读到两个矛盾的版本
- 判据：如果你在 L0 写的内容超过两句话，它大概该在 L1

### 红线 2：禁止只留链接不留内容

规则内容必须实体存在于某层。L0 可以写"详见 X.md#Y"，
但前提是 L0 本身已有一句话陈述这条规则**是什么**。

- 违反后果：AI 只看到链接，不知道规则存在，路径没命中时完全不读
- 判据：把链接删掉，剩下一句话是否还能让 AI 知道"有这堵墙"？能 → 合格

### 红线 3：禁止 L0 膨胀

L0 理念有上限（~12 条以内）。判据：一条规则只影响一个子系统、
不影响其他子系统读者 → 不进 L0。

- 违反后果：L0 超过 ~12 条 → AI 不通读 → 理念形同虚设
- 缓解：每次往 L0 加内容，检查是否有一条旧内容可下沉

### 红线 4：禁止 section 改名而不更新索引

索引表锚点是 section heading。heading 改名 = 锚点断裂 = AI 读到空段。

- 违反后果：AI 按索引加载，拿到的是错的 section 或空
- 判据：section 重命名是 STRUCTURAL 改动，必须走文档影响检查工作流，
  索引表同步更新

### 红线 5：禁止新增代码路径而不登记索引

新增目录或新增导出表面，如果没在索引表加行 → 该路径成为检索盲区。

- 违反后果：新代码无文档保护
- 判据：文档影响检查工作流 Step 3 的断言之一

### 红线 6：被索引的 section 必须自包含

一个 section 被索引表指向，意味着 AI 可能**只读这一段**不读上下文。
如果它依赖前文才能理解，检索就失效。

- 违反后果：AI 读了 section 但理解错误
- 判据：把 section 单独抽出来读，能否独立理解？
  引用其他 section 时用显式锚点（"见 §X"），让 AI 按需追读

### 红线 7：禁止删除 L0 理念或 L1 约束而不检查代码依赖

删前必须确认：是否有代码靠这条理念/约束保护？删除后违反会怎样？

- 违反后果：保护撤除，后续 AI 改代码时写出违规代码
- 判据：删除是 STRUCTURAL 改动，必须走文档影响检查工作流

## 文档影响检查工作流

改完代码后，按此流程判定是否需要更新文档：

### Step 1：结构性分类

判定改动是 LOCAL 还是 STRUCTURAL：

| 信号 | 判定 |
|---|---|
| 改了接口签名、事件 payload、DB schema、导出符号、依赖方向 | STRUCTURAL |
| 同类逻辑存在 ≥2 种写法，本次改动是其中一种（模式漂移机会） | STRUCTURAL |
| 改动跨过了模块边界（events/ 引了 services/、adapters/ 引了业务模块） | STRUCTURAL |
| 新增/删除导出表面（service 导出函数、事件类型、adapter 注册） | STRUCTURAL |
| 改某 service 内部实现、修 bug、调常量 | LOCAL |

- LOCAL → 只查 L2（如有）
- STRUCTURAL → 进入 Step 2

### Step 2：逐层判定

**[L0] 理念是否仍然成立？**

| 触发信号 | 定位 | 动作 |
|---|---|---|
| 代码行为与理念陈述矛盾 | 按约束主题查理念段 | 删/改（走检查工作流，确认是理念错了还是代码错了） |
| 新增了跨子系统设计理念 | 查是否已有同类理念 | 满足全局判据则新增 |
| 理念范围变窄（原全局变成单子系统） | 不再满足 L0 判据 | 下沉到 L1 |

不需要更新的信号：改了某 service 内部实现 → L0 理念"service 无框架依赖"不变。

**[L1] 契约是否仍然准确？**

| 触发信号 | 定位（按改动性质→契约维度） | 动作 |
|---|---|---|
| 接口签名变化 | §接口定义 / §关键接口 | 更新接口表 |
| 新增/删除导出符号 | §接口定义 | 增删条目 |
| 依赖方向被触及 | §依赖方向 | 先修代码再同步文档 |
| 新增事件/subscriber | §事件定义 / §数据流 / §架构图 | 增加条目 |
| 扩展点变化 | §新增事件 / §新增Agent类型 | 更新步骤 |
| 概念定义变化 | §概念表（如有） | 更新定义+检查所有引用处术语 |

**[L2] 实现是否仍然如此？**

| 触发信号 | 定位（按操作场景→实现段） | 动作 |
|---|---|---|
| 路径变化 | §目录结构 | 更新路径 |
| 函数/变量改名 | §模式 / §迁移清单 | 更新引用 |
| 新增实现模式 | §模式 | 增加条目 |
| 删除实现路径 | 对应段 | 删条目或标废弃 |
| 行为细节变化 | 对应描述 | 更新 |

### Step 3：索引表一致性

- 改动涉及的路径在索引表有对应行吗？没有 → 补行
- section heading 改名了吗？→ 同步索引锚点
- 新增代码路径有索引行吗？→ 没有 → 补行

## Section 质量约束

### 自包含性

被索引的 section 必须能在不读其他 section 的情况下独立理解。
检查方法：把 section 单独抽出来读，能否独立理解？
引用其他 section 时用显式锚点（"见 §X"），而非隐式依赖前文。

### 命名稳定性

section heading 是索引锚点。重命名 = 断链。
约束：section 重命名时必须同步更新索引表。

## 文档粒度判据

### 新建文档

- 该子系统有 ≥3 个 L1 级约束（契约段）需要独立论述
- 或有 ≥1 个 L0 理念的完整实现需要展开
- 且现有文档无法自然容纳（不属于任何现有子系统的边界）

### 合并到现有文档

- 内容 <50 行且与现有文档同一子系统
- 或内容是现有文档的 L2 细节

### 拆分文档

- 单篇 >400 行且 L1/L2 混杂无法内部分节
- 或一个子系统的 L1 契约和 L2 实现可独立服务于不同读者

## 文档废弃判据

| 场景 | 动作 |
|---|---|
| 代码删除 + 概念废弃 | 删除文档，更新索引表 |
| 代码删除 + 概念仍存（被其他实现替代） | 保留概念段，更新实现段，标注迁移路径 |
| 渐进废弃 | 文档头部标记 `<!-- status: deprecated, replaced-by: X.md -->` |

## 代码事实标记

文档中引用可机器核对的代码事实时，用 `code-fact` 注释标记：

```markdown
<!-- code-fact: adapter types = registry.listTypes() -->
当前已注册: claude, codex, gemini, shell
```

更新文档时，凡有 `code-fact` 标记的段，必须重新核对代码事实。
标记规则：**所有枚举性事实（类型列表、事件列表、路由列表）必须标记**。
非枚举性描述（设计模式、架构方向）不标记。

## 文档间依赖登记

每篇 `architecture/*.md` 文件头部声明它依赖的其他文档 section：

```markdown
<!-- depends-on: event-bus.md#事件定义, adapters.md#Adapter接口 -->
```

文档 A 改了某 section → 查所有 `depends-on` 引用了该 section 的文档，
它们需要核对一致性。
